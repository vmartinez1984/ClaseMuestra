App de banco

Vistazo

Será una aplicación de tipo banco recortada, donde tendremos los siguientes componentes

Registro
Se registra el cliente con los datos básicos generales, nombres, apellidos, correo, telefono, rfc y contraseña
Inicio de sesión
Para ingresar a la aplicación es necesario porporcionar un inicio de sesión, este será por medio de usuario y contraseña
Datos generales de la cuenta
Aqui se mostrará el total de la cuenta, el numero de cuenta, el número de clabe
Detalles de los movimientos
Lista de cantidad, referencia, fecha del movimiento, tipo de movimiento, concepto.
Transferencia
Cantidad, nombre del beneficiario, clabe, concepto
Instrucciones para realizar depositos
Se mostrara el número de cuenta, un código de barras y las instrucciones para el deposito. En este punto se consumira el servicio por medio de postman o una pagina que simulara la famosa cadena de tiendas, el cual recibira el dinero para que se aplicado a la cuenta de ahorro.

Donde consideraremos:
Los repositorios es el origen de datos una db Sql, db NoSql, archivo txt, archivo csv, archivo tsv incluso otro servicio.
Los almacenes se utilizan para guardar archivos, que pueden ser la documentación, imagenes, videos. Esto puede ser una almacen de objetos como Alfresco, S3 de Amazon, por mencionar un par.
Los servicios se usa para el procesamiento de datos, como puede ser extracción ocr, reconocimiento facail, analisis de imagenes, generación de clabes, generación de RFC, etc.

¿Como se organiza el proyecto?

Utilizaremos principios SOLID, aunque suene muy complejo en relidad ya ocupaste algunos conceptos anteriormente en tus practicas de MVC y Web services. Los postfijos, prefijos y/o convenciones se haran en ingles, el nombre de las clases serán en español. Pero usted puede hacerlas como guste, solo mantenga uniformidad en los nombres y sus propias convenciones. Si usted usa ClienteModel, ClienteService en algunas de sus clases y por ahi aparezca ClientViewModel, sera observado en su code review para corrección o su penalización en caso de no hacerlo.

Organice sus carpetas y clases, como se muestra a continuación

Paginas
Aqui iran todas las pages que se crean en el proyecto con el postfijo Page, que es la componenete donde tiene interacción el cliente o usuario.
ej. RegistroDeClientePage
Modelos
Son las clases que tienen su correspondencia con los campos de la página y deben de tener el postfijo Model
ej. ClienteModel
ModeloVistas
Son los componentes relacionados con el patron MVVM y deben de tener el psotfijo ModelView, esta clase tiene la responsabilidad de la comunicación de la vista es decir la Page con el Modelo
ej. ClienteModelView
Servicios
Es la carpeta que se encargara de procesar y enviar los datos al web service. Debe de llevar el postfijo Service. Implementacion de las interfaces.
ej. ClienteService
Interfaces
Aqui iran todas las interfaces, las cuales seran inyectadas en el MauiProgram. Deben de llevar el prefijo I
ej. IClienteService
